atto — Streaming and Cooperative Servers

This arc runs from 07-Jun-2026 to 20-Jun-2026. It has two halves. The first backports the streaming and non-blocking work that had been developed in the Yioop project so that atto's own servers no longer block a whole process on a single slow read or write. The second reworks the servers' event loops around PHP Fibers so that one atto process can carry several long-lived transfers cooperatively rather than serially. Both were driven by the same need seen in the pure-PHP HTTP/3 work: a single-file, single-process server has to yield the CPU carefully or one client stalls the rest. Commits below are grouped by the two halves.

  1. ✓ Non-blocking streaming, backported from Yioop (07-Jun to 13-Jun).
    1. ✓ Remove the stream-blocking behaviour in both MailSite and WebSite, and make the FileStorage backend more efficient so a large body is neither read nor written in one blocking call (a48193d).
    2. ✓ Backport the streaming tweaks worked out in Yioop so the two projects share the same streaming shape (b788bed).
    3. ✓ Better streaming HTTP/2 support in WebSite, then a memory-bounded HTTP/3 generator-based streaming path so a response body is produced and sent in bounded chunks rather than materialised whole; the H3 stream is refilled on the inbound path so throughput does not stall waiting for the next chunk (ff8317b, fd562c3, ecef6df).
    4. ✓ Wire the streaming case into the benchmarks dashboard and refresh examples 15 (Streaming) and 18 (HTTP Benchmarks) to exercise it (61c4647, 9ce4685).
    5. ✓ Fix a fractional-timeout calculation bug in WebSite and a select spin in GopherSite; a MailSite spin fix landed alongside the WebSite streaming work (9ded403, db462ca, 532166d).
  2. ✓ Cooperative event loops with PHP Fibers (20-Jun).
    1. ✓ Rework the event loops so a long-running server step (a mail delivery, an FTP transfer) runs inside a Fiber that yields back to the loop instead of blocking it, letting one process interleave several transfers (1029c8f).
    2. ✓ Make MailSite cooperative, then FtpSite: a cooperative setup path, transfer pumps that move bytes a chunk at a time between yields, and a fix to the FTP web UI upload (3514b4b, 882f939, c090140, b39e214).
    3. ✓ A benchmark defer case for the fourth slice so the cooperative path is measured against the blocking one (8de775a).
    4. ✓ Add example 26, Cooperative Fiber Primitives, showing the raw yield/resume primitives, with a web rewrite and code toggles so a reader can watch the loop hand off between tasks (f5682f8, 72b9de6, 55eda06).
  3. ✓ Follow-up fix (02-Jul). A set_error_handler memory leak: the handler was only ever pushed onto the internal stack and never popped, so long-lived server processes grew over time. Fixed so each temporary handler is restored (e89641d). This same shape was later removed outright on the Yioop side of the streaming write path.
