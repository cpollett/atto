atto — CalDAV Server

Started 05-Jul-2026. CalDAV (RFC 4791) is calendaring layered on top of WebDAV: a calendar is a folder (collection), each event or task is a small file inside it in the iCalendar text format (RFC 5545), and a couple of extra HTTP verbs let a client create calendars and ask time-range questions about them. atto's WebSite already routes the WebDAV verbs and already ships a folder-backed WebDAV share in example 14; this arc adds the two verbs CalDAV needs to that same routing, then puts the calendar behaviour in a reusable CalDav class in src/ that an example wires into a WebSite — the same folder-backed shape as the WebDAV example, so the pattern a reader already knows carries straight over, and written so the Yioop project can reuse the same class against its own WebSite. The goal is a share a real calendar client (Apple Calendar, Mozilla Thunderbird, the Android DAVx5 app) can be pointed at and left to sync on its own, while keeping atto's no-config, single-file character: nothing Yioop-specific, no framework, no build step.

What CalDAV adds on top of WebDAV

So the plan below reads plainly, here is the whole of what is new versus the existing WebDAV share, in one place:

  • Two new HTTP verbs: MKCALENDAR (make a calendar collection) and REPORT (a POST-like read that carries an XML query body and returns a 207 multi-status answer).
  • A calendar collection is just a folder that additionally reports the <C:calendar/> resource type and a supported-component list (VEVENT, VTODO) in its properties.
  • An event is just a .ics file (one iCalendar VEVENT/VTODO object) whose bytes the client writes with PUT and reads with GET, exactly as in the WebDAV example.
  • Every resource carries an ETag, and each collection a change tag, so a client can tell what changed without re-reading everything; conflicting writes are caught with If-Match / If-None-Match.
  • A small discovery path so a client given only the base URL can find the calendar by itself, rather than the user typing a collection URL.
  1. → Add the two CalDAV verbs, REPORT and MKCALENDAR, to atto's WebSite routing, so a CalDAV handler is an ordinary route callback the way the WebDAV verbs already are. Concretely this touches the places that each carry an explicit method list:
    1. The $routes method table (add REPORT and MKCALENDAR keys).
    2. The two request-line method patterns — the one in checkHttpType and the one that parses the first request line — so a request naming either verb is accepted rather than answered 400.
    3. Raise LEN_LONGEST_HTTP_METHOD from 9 to 10, since MKCALENDAR is ten characters, and correct that constant's docblock, which today reads “length 7” and is already wrong for the 9-character PROPFIND.
    4. Confirm the existing WebDAV example and its tests still pass unchanged, since this step only widens the set of accepted verbs.
  2. Put the calendar behaviour in a reusable CalDav class in src/CalDav.php, and have the example wire it into a WebSite. This class is the calendar endpoint the atto example uses, and is written so the Yioop project can reuse it against its own WebSite subclass. It stays self-contained in atto's style: the seekquarry\atto namespace, no framework or config references, no Composer, and a docblock on the class and on every method.
    1. Composition, not a subclass. The constructor takes the WebSite (or subclass) instance it serves, the directory the calendars live in, a URL prefix (default /calendars), and an authenticator callable of the shape example 14 already uses (return true to proceed, or emit a 401 and return false). One register method adds every calendar route to the site with addRoute, so each handler is an ordinary atto route. Taking the site rather than extending it is what lets Yioop attach the calendar to the WebSite subclass it already runs.
    2. All calendar logic lives on the class so the example and Yioop run the same code: the per-verb handlers, the iCalendar reading, the ETag computation, and the multi-status XML building are methods, each overridable so a reuser can swap storage or discovery without rewriting the rest.
    3. Folder-backed storage by default — a calendar is a folder, an event a .ics file — dropping in over a directory the way the WebDAV example does, with the same resolve-and-contain path guard so a request cannot escape the base.
    4. A small built-in reader for the request bodies, covering just the fixed CalDAV vocabulary (the few elements in a calendar-query, calendar-multiget, PROPFIND, and MKCALENDAR body), rather than depending on ext-dom. atto's WebDAV code already builds its XML by hand; this keeps CalDav.php pure-PHP and working where the DOM extension is absent, and the vocabulary is small and closed enough to read directly.
  3. The verbs the CalDav class answers over its prefix:
    1. OPTIONS advertising DAV: 1, 2, 3, calendar-access and listing REPORT and MKCALENDAR in Allow, so a client knows it is talking to a CalDAV server.
    2. MKCALENDAR creating a calendar collection (a folder) and recording its display name and supported component set from the request body.
    3. PROPFIND extended over the WebDAV one to report the calendar properties on collections (resourcetype including <C:calendar/>, supported-calendar-component-set, a change tag) and an ETag on each .ics resource.
    4. PUT / GET / DELETE of .ics resources, returning and honouring ETags (If-Match for a safe overwrite, If-None-Match: * for a safe create).
    5. REPORT for the two calendaring reports clients rely on: calendar-query (filter by component type and time range) and calendar-multiget (return a named set of resources), each answered as a 207 multi-status carrying the matching calendar data.
  4. Give CalDav just enough iCalendar (RFC 5545) reading to serve calendar-query: a method that pulls the component type and the DTSTART/DTEND (or DURATION) out of a stored .ics file so the time-range filter can include or exclude it. The class stores the file's bytes verbatim on PUT and hands them back verbatim on GET and in a report — it reads the calendar data but does not rewrite it, which keeps the round trip lossless and the parsing surface small.
  5. Client autodiscovery on the class, so a real client needs only the base URL:
    1. A /.well-known/caldav bootstrap (RFC 6764) that redirects a probing client to the service.
    2. The principal properties a client walks to find a calendar: current-user-principal (RFC 5397) and calendar-home-set, answered by PROPFIND so the client can go base URL → principal → calendar home → calendar without the user pasting a collection URL.
    3. A cheap change signal for sync — a collection change tag (CTag) reported on the calendar, so a client polls one property to learn whether anything changed before listing the collection.
  6. Tests under tests/caldav/test_*.php, following atto's standalone-script contract (assert helper, Tests run: / Tests passed: summary, non-zero exit on failure, each check well under the I/O-flag threshold). Because the logic lives on the CalDav class as methods, the tests exercise it directly without a live server: the iCalendar time-range filter (does this event fall in this window), the ETag computation, the small body parser, and the 207 XML building each get direct assertions on known input and output.
  7. Documentation to match the other examples: a file header docblock on CalDav.php, and an example whose index.php constructs a WebSite, hands it to CalDav with a calendar directory and an authenticator, and adds a browser landing page giving a self-contained curl round trip (MKCALENDAR a calendar → PUT an event → REPORT a calendar-query → GET it back → DELETE it), plus a “how to connect” pointer for pointing Apple Calendar, Thunderbird, or DAVx5 at the share. Add the new example to the examples index and the README example tour.

Open decisions to settle before coding

  • Example placement and numbering. The CalDAV example belongs next to WebDAV (example 14). Inserting it as example 15 means renumbering 15–27 upward, the same churn the WebDAV back-port just did; appending it as the next free number avoids the renumber but separates it from the WebDAV example it builds on. Chris to pick which.
  • Request-body XML: built-in reader versus ext-dom. The plan reads the small, fixed CalDAV request bodies with a built-in parser, to stay pure-PHP and to run where the DOM extension is absent (as it is in the sandbox PHP build used here). If leaning on ext-dom/SimpleXML where present is preferred, the class can use that instead; the choice is contained to one reading method.
  • Scheduling scope. This arc targets calendar storage and querying (RFC 4791) only, not the scheduling extension (RFC 6638, invitations and free/busy between users). The same REPORT verb and the same folder-backed pattern would later carry a CardDAV (contacts) example too; both are out of scope here and noted only so the boundary is explicit.
