atto — CalDAV Server

Started 05-Jul-2026. CalDAV (RFC 4791) is calendaring layered on top of WebDAV: a calendar is a folder (collection), each event or task is a small file inside it in the iCalendar text format (RFC 5545), and a couple of extra HTTP verbs let a client create calendars and ask time-range questions about them. atto's WebSite already routes the WebDAV verbs and already ships a folder-backed WebDAV share in example 14; this arc adds the two verbs CalDAV needs to that same routing, then puts the calendar behaviour in a reusable CalDav class in src/ that an example wires into a WebSite — the same folder-backed shape as the WebDAV example, so the pattern a reader already knows carries straight over, and written so the Yioop project can reuse the same class against its own WebSite. The goal is a share a real calendar client (Apple Calendar, Mozilla Thunderbird, the Android DAVx5 app) can be pointed at and left to sync on its own, while keeping atto's no-config, single-file character: nothing Yioop-specific, no framework, no build step.

What CalDAV adds on top of WebDAV

So the plan below reads plainly, here is the whole of what is new versus the existing WebDAV share, in one place:

  • Two new HTTP verbs: MKCALENDAR (make a calendar collection) and REPORT (a POST-like read that carries an XML query body and returns a 207 multi-status answer).
  • A calendar collection is just a folder that additionally reports the <C:calendar/> resource type and a supported-component list (VEVENT, VTODO) in its properties.
  • An event is just a .ics file (one iCalendar VEVENT/VTODO object) whose bytes the client writes with PUT and reads with GET, exactly as in the WebDAV example.
  • Every resource carries an ETag, and each collection a change tag, so a client can tell what changed without re-reading everything; conflicting writes are caught with If-Match / If-None-Match.
  • A small discovery path so a client given only the base URL can find the calendar by itself, rather than the user typing a collection URL.

    • The $routes method table (add REPORT and MKCALENDAR keys).
    • The two request-line method patterns — the one in checkHttpType and the one that parses the first request line — so a request naming either verb is accepted rather than answered 400.
    • Raise LEN_LONGEST_HTTP_METHOD from 9 to 10, since MKCALENDAR is ten characters, and correct that constant's docblock, which today reads “length 7” and is already wrong for the 9-character PROPFIND.
    • Confirm the existing WebDAV example and its tests still pass unchanged, since this step only widens the accepted verbs.

    Done: the two verbs were added to the routes table, the length constant (9→10, docblock corrected), the checkHttpType sniff, and the request-line parser. The HTTP/2 path needed no change — it passes :method through. A unit test, tests/caldav/test_caldav_verbs.php, covers registration, dispatch, the length budget, the sniff, and a full request-line parse, each with a bogus-verb control; the whole suite stays green.

  2. ✓ Put the calendar behaviour in a reusable CalDav class in src/CalDav.php, and have the example wire it into a WebSite. This class is the calendar endpoint the atto example uses, and is written so the Yioop project can reuse it against its own WebSite subclass. It stays self-contained in atto's style: the seekquarry\atto namespace, no framework or config references, no Composer, and a docblock on the class and on every method.
    • Composition, not a subclass. The constructor takes the WebSite (or subclass) instance it serves, the directory the calendars live in, a URL prefix (default /calendars), and an authenticator callable of the shape example 14 already uses (return true to proceed, or emit a 401 and return false). One register method adds every calendar route to the site with addRoute, so each handler is an ordinary atto route. Taking the site rather than extending it is what lets Yioop attach the calendar to the WebSite subclass it already runs.
    • All calendar logic lives on the class so the example and Yioop run the same code: the per-verb handlers, the iCalendar reading, the ETag computation, and the multi-status XML building are methods, each overridable so a reuser can swap storage or discovery without rewriting the rest.
    • Folder-backed storage by default — a calendar is a folder, an event a .ics file — dropping in over a directory the way the WebDAV example does, with the same resolve-and-contain path guard so a request cannot escape the base.
    • A small built-in reader for the request bodies, covering just the fixed CalDAV vocabulary, rather than depending on ext-dom. atto's WebDAV code already builds its XML by hand; this keeps CalDav.php pure-PHP and working where the DOM extension is absent.

    Done: src/CalDav.php attaches to a WebSite by composition and serves the calendar lifecycle over its prefix: OPTIONS (advertises DAV: 1, calendar-access), MKCALENDAR, PROPFIND (a calendar reports as a calendar with its component set and change tag; an event reports its ETag, type, size, and modified time), and PUT, GET, DELETE of .ics events with the ETag preconditions If-Match and If-None-Match. Entity and change tags use SHA-256. Each handler's status line takes its protocol from the request, so the same code serves HTTP/1.1, HTTP/2, and HTTP/3. Tests in tests/caldav/test_caldav_class.php (22 checks) cover the pure helpers and the full lifecycle over a scratch folder.

  3. ✓ Add the REPORT verb and the iCalendar reading it needs, the one part of the calendar lifecycle not yet in place. The other verbs shipped with the class above; this item is what lets a client ask a calendar questions rather than only list it.
    • calendar-multiget: the body names a set of event hrefs and the reply returns each one's ETag and calendar data in a 207 multi-status, so a client fetches many events in one round trip. This needs no date reading — it returns named events as stored.
    • calendar-query: the body carries a filter by component type (VEVENT, VTODO) and an optional time range, and the reply returns the matching events. This is the report a client uses to load a month or a week.
    • The iCalendar (RFC 5545) reading calendar-query needs: a method that pulls the component type and the DTSTART/DTEND (or DURATION) out of a stored .ics file, and an overlap test that decides whether an event falls in the queried window. The stored bytes are still returned verbatim; the reading only decides what to include.
    • Wiring: register the REPORT route, and add it to the OPTIONS Allow list now that it is answered.
    • Tests ride in the same patch: the time parsing and overlap test as pure checks on known input, and each report driven over a scratch folder with events at known times.

    Done: handleReport answers calendar-multiget (returns each named event's ETag and data, a not-found entry for a missing one) and calendar-query (filters by component type and an optional time range). The iCalendar reader — parseIcalTime, parseDuration, icalComponents, icalEventRange, and an overlap test — reads times as UTC and covers the common VEVENT cases, not the full RFC 4791 §9.9 matrix or named-zone conversion; the REPORT route is registered and advertised in OPTIONS. Tests in tests/caldav/test_caldav_report.php (20 checks).

  4. ✓ Client autodiscovery on the class, so a real client needs only the base URL:
    • A /.well-known/caldav bootstrap (RFC 6764) that redirects a probing client to the service.
    • The principal properties a client walks to find a calendar: current-user-principal (RFC 5397) and calendar-home-set, answered by PROPFIND so the client can go base URL → principal → calendar home → calendar without the user pasting a collection URL.
    • A cheap change signal for sync — the collection change tag (CTag) already reported on a calendar, so a client polls one property to learn whether anything changed before listing.
  5. ✓ Documentation and a runnable example: a file header docblock on CalDav.php (already in place), and an example whose index.php constructs a WebSite, hands it to CalDav with a calendar directory and an authenticator, and adds a browser landing page giving a self-contained curl round trip (MKCALENDAR a calendar → PUT an event → REPORT a calendar-query → GET it back → DELETE it), plus a “how to connect” pointer for Apple Calendar, Thunderbird, or DAVx5. Add the example to the examples index and the README tour.

    Done: examples/15 CalDAV makes a WebSite, hands it to CalDav, calls register(), and seeds a demo calendar. The browser page is a live calendar: on a timer it sends the demo calendar a calendar-query REPORT and lists the events that come back, so a change shows up on the next refresh. Links on the page add fixed sample events with a CalDAV PUT, and a portable printf-into-a-file plus curl --data-binary @file block adds one from the command line the same way (no $'…', so it runs in tcsh as written); a “connect a real client” pointer covers Apple Calendar, Thunderbird, and DAVx5. The demo runs open (no authenticator) to keep the page and curl friction-free; a note points at example 14 for the authenticator shape. Open decisions to settle before coding

  • Example placement and numbering. Resolved: the CalDAV example was inserted at 15, next to WebDAV; examples 15–27 were renumbered to 16–28, and their inter-example references and the README tour were updated to match.
  • Request-body XML: built-in reader versus ext-dom. The plan reads the small, fixed CalDAV request bodies with a built-in parser, to stay pure-PHP and to run where the DOM extension is absent (as it is in the sandbox PHP build used here). If leaning on ext-dom/SimpleXML where present is preferred, the class can use that instead; the choice is contained to one reading method.
  • Scheduling scope. This arc targets calendar storage and querying (RFC 4791) only, not the scheduling extension (RFC 6638, invitations and free/busy between users). The same REPORT verb and the same folder-backed pattern would later carry a CardDAV (contacts) example too; both are out of scope here and noted only so the boundary is explicit.
