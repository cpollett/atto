atto — CardDAV Server

Started 07-Jul-2026. CardDAV (RFC 6352) is the address-book sibling of CalDAV: same WebDAV substrate, but a collection holds contacts instead of events. An address book is a folder (collection), each contact is a small file inside it in the vCard text format (RFC 6350), and the same read verbs let a client list a book and ask property questions about it. atto already routes every WebDAV verb CardDAV needs — MKCOL and REPORT both route today (the latter was added for CalDAV) — and already ships the folder-backed collection pattern twice, in the WebDAV share (example 14) and the CalDAV server (example 15). This arc puts the contact behaviour in a reusable class in src/ that an example wires into a WebSite, the same shape as those two, and written so the Yioop project can reuse it against its own WebSite. The goal is an address book a real contacts client (Apple Contacts on macOS and iOS, Mozilla Thunderbird, the Android DAVx5 app) can be pointed at and left to sync on its own, while keeping atto's no-config, single-file character: nothing Yioop-specific, no framework, no build step.

What CardDAV adds on top of what is already here

So the plan below reads plainly, here is the whole of what is new versus the CalDAV server, in one place. The two protocols share a large WebDAV and collection core; the list is short because most of that core already exists.

  • No new HTTP verbs. An address book is created with an extended MKCOL (RFC 5689) whose body sets the collection's resource type, where CalDAV used a dedicated MKCALENDAR; both MKCOL and REPORT already route.
  • An address book is a folder that reports the <CARD:addressbook/> resource type and a supported-address-data media type (text/vcard) in its properties, where a calendar reports <C:calendar/> and a component list.
  • A contact is a .vcf file (one vCard object) whose bytes the client writes with PUT and reads with GET, exactly as an event is a .ics file.
  • The REPORT query is different in kind: CalDAV's calendar-query filters components by a time range; CardDAV's addressbook-query filters contacts by matching text against vCard properties (name, e-mail, and so on). The multiget report and the ETag/change-tag machinery carry over unchanged.
  • Discovery uses /.well-known/carddav and reports an addressbook-home-set, the direct analogue of CalDAV's well-known path and calendar-home-set.
  1. ✓ The contact-server class: construction, route registration, and the WebDAV verbs an address book answers. Nearly all of this is the same collection machinery the CalDAV class already runs; the address-book-specific parts are the resource type, the supported-address-data property, the text/vcard content type, and creation by extended MKCOL.
    • Construct against a WebSite, a contacts directory, a route prefix, and an optional authenticator, then register() the routes, the same constructor shape as the CalDAV class.
    • Answer OPTIONS (advertising addressbook-access), PROPFIND (reporting an address book, a plain collection, or a contact), GET, PUT, and DELETE, with the same path-containment guard, SHA-256 ETags, and If-Match / If-None-Match preconditions the CalDAV class uses.
    • Create an address book from an extended MKCOL whose body carries a resource type including addressbook and an optional display name, writing the same kind of hidden metadata file the CalDAV class writes for a calendar.
    • Tests in the same patch: create a book, PUT and GET a contact, PROPFIND the book at depth 0 and 1, and the ETag preconditions, each over a scratch folder.

  2. ✓ The REPORT verb and the vCard reading it needs. Multiget carries over from CalDAV almost verbatim; the query is the new work.
    • addressbook-multiget: return each named contact's ETag and data, and a not-found entry for a missing one — the same shape as calendar-multiget.
    • addressbook-query: filter contacts with a prop-filter and a text-match (match a vCard property such as FN or EMAIL by equals, contains, starts-with, or ends-with), combined under an anyof or allof test, and return the matches' data.
    • A small vCard reader for the query: pull a named property's value from a contact's lines, enough for the common unfolded PROP;PARAMS:VALUE form, mirroring the scope note the CalDAV query carries for iCalendar.
    • Tests: the property match as pure checks on known vCards, and each report driven over a scratch folder of contacts with known names and addresses.

    Done: handleReport answers addressbook-multiget (each named contact's ETag and data, a not-found entry for a missing one) and addressbook-query (filter contacts by a prop-filter under an anyof or allof test). A prop-filter judges a contact by a text-match (equals, starts-with, ends-with, or contains, case-insensitively), by is-not-defined (the property absent), or by presence when it carries neither. The vCard reader pulls a named property's values from the common unfolded PROP;PARAMS:VALUE lines; folding, param-filter, collation negotiation, and restricting the returned address-data stay out of this first cut, as the plan's scope note says. Tests in tests/carddav/test_carddav_report.php (13 checks). Verified live: an FN-contains query returns only the matching contact, and a multiget returns the named contacts with a not-found for a gap.

  3. ✓ Client autodiscovery, so a real client needs only the base URL.

    • Report current-user-principal and addressbook-home-set on the collection and address-book responses, both pointing at the route prefix by default and overridable for a project (such as Yioop) with per-user homes.

    Done: handleWellKnown answers /.well-known/carddav with a 301 to the service root (routed by register() for GET and PROPFIND, ungated), and register() gained a $with_well_known toggle (on by default) so a project that owns /.well-known can skip it. PROPFIND on a book or on the folder that holds the books now carries current-user-principal and addressbook-home-set through a principalProps method, both pointing at the route prefix and overridable for a project (such as Yioop) with per-user homes. Discovery tests added to tests/carddav/test_carddav_class.php (now 21 checks). Verified live: the well-known path redirects to /addressbooks/, and PROPFIND reports both discovery properties.

Open decisions to settle before coding

  • How the two servers share code. CalDAV and CardDAV share a large WebDAV and collection core (construction, containment, ETags, PROPFIND, the GET/PUT/DELETE verbs, multiget, discovery); they differ in the resource type, the well-known path, the home-set property, the create mechanism, the content type, and the REPORT query. Two shapes both fit atto's rule that a src/ file is self-contained (no cross-file requires or inheritance): (a) a standalone CardDav.php that reimplements the shared core, self-contained and touching neither the shipped CalDav.php nor Yioop, at the cost of duplicating roughly two dozen methods — the way each other src/ server stands on its own today; or (b) one file holding an abstract collection base plus both CalDav and CardDav subclasses, the way DnsSite and FtpSite keep an abstract base with its concretes, which removes the duplication but restructures the shipped CalDav.php, changes the file the CalDAV example and Yioop require, and puts two protocols in one file. Shape (a) keeps shipped code untouched and matches how the other servers are organised; shape (b) is the one with no duplication. If (b) is chosen, item 1 above absorbs extracting the base and retrofitting CalDav onto it, guarded by the existing CalDAV tests. Resolved: the standalone shape — src/CardDav.php reimplements the shared core, leaving the shipped CalDav.php and Yioop untouched, in keeping with how the other servers each stand on their own.
  • Address-book creation. RFC 6352 has no MKADDRESSBOOK verb; a book is made with an extended MKCOL (RFC 5689) carrying a resource type, which is what Apple Contacts and DAVx5 send. The plan implements that. The example additionally seeds a demo book at start-up so its page shows contacts before anything is created, the way the CalDAV example seeds a demo calendar.
  • vCard query scope. The first cut handles the common addressbook-query: prop-filter with a text-match and match type, under anyof or allof. Left for later, and named in a scope note the way the CalDAV REPORT names its own limits: param-filter, collation negotiation, restricting the returned address-data to named properties, and vCard line folding. If a wider first cut is wanted, say which parts.
  • Example placement and numbering. CardDAV belongs next to CalDAV (example 15). Inserting it as example 16 renumbers 16–28 upward, the same churn the CalDAV insert just did (renames plus a handful of inter-example reference and README edits); appending it as the next free number avoids the renumber but separates it from the CalDAV example it mirrors. The CalDAV arc chose the insert for coherence; this arc can follow that or not. Resolved: inserted at 16 next to CalDAV; examples 16–28 were renumbered to 17–29, and their inter-example references and the README tour updated to match.
