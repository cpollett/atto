atto — H3 Listener Hotpath Benchmarks

Started 07-Jul-2026. The Post v2 note h3-status-and-bottlenecks.html lists, per method, where the pure-PHP HTTP/3 listener spends its time on a 1 MiB response, but it reported those numbers without a test anyone can run to reproduce or inspect them. This arc builds that missing measurement: for each hotpath method the note names, a benchmark that calls that one method in isolation over many iterations and reports its per-call cost. The point is twofold: reproduce the note's per-method figures so they can be inspected directly, and give each method a stable baseline so a later code tweak can be measured against it — run the benchmark, change the method, run it again, and read the difference. These are benchmarks, not the pass/fail tests: they stand up no H3 website and open no socket, they exercise a single method with representative inputs, and they are timed rather than asserted.

Two things already noticed while reading the code, worth carrying into the work rather than trusting the note blindly. The substr-chunking fix the note lists as a future win is already in place: QuicStream advances a send_buf_off offset instead of mutating the buffer, so takeForFrame's benchmark will measure the current offset-based cost, not the old O(n^2) shift. And the QuicConnection constructor takes a certificate and key, so the connection-stateful methods need a throwaway cert plus a representative response queued before they can be called; that set-up is the bulk of the work for the last two items.

  1. ✓ The benchmark runner and the AEAD benchmarks — the largest single cost (~36 ms, about a third of a /big response) and the cleanest to isolate.
    • A small bench helper that warms a method up, times a fixed number of calls with hrtime, takes the median over a few repetitions to smooth noise, and reports nanoseconds per call and calls per second. A bench_all.php that runs the benchmarks and prints one table.
    • Benchmarks for seal, open, and headerProtectionMask on a real key set built from QuicPacketKeys::fromInitialDcid (AES-128-GCM, the cipher the note measured), with a ~1100-byte payload and a 16-byte header-protection sample, the sizes the note used.
    • Confirm the per-call figures land near the note's ~4.2, ~8, and ~2.6 us; where they differ, say so plainly, since the box and the openssl build both move these numbers.

    Done: tests/h3bench/ holds the benchmarks. measure.php is the shared helper — it warms a call, times a fixed number of iterations with hrtime, reports the median per-call time over several repetitions, and prints a row in microseconds per call and calls per second. bench_aead.php builds a real AES-128-GCM key set through QuicPacketKeys::fromInitialDcid, checks the inputs form a valid seal/open pair and a five-byte mask before timing, then benchmarks seal, open, and headerProtectionMask on a 1100-byte payload, with an empty-closure row so the wrapper overhead is visible. bench_all.php runs each bench_*.php in its own process. The bench_ prefix and the measure.php name both stay clear of test_*.php, so tests/run_all.php does not pick them up; it stays 16 files green. Measured on this box: seal ~3.0 us/call, open ~3.0 us/call, headerProtectionMask ~1.1 us/call, empty-closure overhead ~0.02 us. Same order as the note's ~4.2, ~8, and ~2.6 us; faster here, since this is a different CPU and openssl build, and open sits level with seal rather than above it (the note's open figure came from a small sample). Run with php tests/h3bench/bench_aead.php or php tests/h3bench/bench_all.php.

  2. ✓ Frame and packet encode/decode benchmarks.
    • QuicFrame::encode on a STREAM frame (its hot case, which dispatches into encodeStream), and decodeAll over a buffer of the frames a response and its ACKs carry.
    • QuicPacket::encodeShort and decodeShort over a short-header packet, using the key set from item 1 so the AEAD and header-protection steps they call are exercised as they are in the real path.

    Done: tests/h3bench/bench_codec.php benchmarks QuicFrame::encode on a 1100-byte STREAM frame, the same frame through encodeStream reached by a reflection closure (so the two differ only by the type switch), decodeAll on the ACK-plus-PING buffer an ACK packet carries, and the short-header pair encodeShort and decodeShort on the item-1 key set, checking each input is valid before timing. Measured on this box: encode ~0.42 us/call, encodeStream ~0.42 us/call, decodeAll ~1.9 us/call, encodeShort ~6.0 us/call, decodeShort ~6.5 us/call. The one result worth flagging: encode and encodeStream come out the same to within noise, so the note's idea that inlining encodeStream to skip the switch saves about a microsecond does not hold here — the switch costs nothing measurable, and that ~0.42 us is the varint writes, the concat, and the payload copy. encodeShort and decodeShort land near their AEAD parts (~3 us seal or open plus ~1.1 us mask) plus the header assembly. Run with php tests/h3bench/bench_codec.php.

  3. ✓ Stream-chunking and flush-loop benchmarks, the first that need object state stood up.
    • QuicStream::takeForFrame over a 1 MiB send_buf, drained in the frame-sized chunks a /big response uses, to measure the current offset-based cost.
    • QuicConnection::flushStreams over a connection with a 1 MiB response queued on a stream, so the flush loop's per-frame chunk-encode-queue cost is measured on a realistic response.

    Done: tests/h3bench/bench_flush.php times takeForFrame both as one chunk and as a full 1 MiB drain, and flushStreams draining that response in one call. The connection is built the way the http3 tests build one — ReflectionClass without the constructor, with only the application keys and the stream map set — so no certificate is needed; that turned out cleaner than the throwaway-cert idea this item first sketched. Measured on this box: takeForFrame ~0.29 us for one chunk; the full 1 MiB drain ~311 us total, ~0.33 us/frame across ~954 frames; flushStreams ~1.4 ms for the whole response, ~1.45 us/frame. The drain being linear is the point: the note records the send buffer once shifting on every drained frame, an O(n^2) ~47 ms cost for a single 1 MiB response; the offset-based buffer brings that same drain to ~0.3 ms here, so that fix is in and working. flushStreams' extra ~1.1 us/frame over the raw drain is the STREAM-frame encode and the queue push. Run with php tests/h3bench/bench_flush.php.

    Scope change carried into item 4: QuicConnection::emit moved out of this item. emit reads far more connection state than flushStreams — the send queue, the per-space packet-number and sent-packet maps, congestion and RTT fields, the peer CID — and it calls the loss-detection timer and the sent-packet recorder, the same whole-connection machinery item 4's setLossDetectionTimer and processAck need. So emit belongs with them, sharing one connection set-up, rather than with the stream-level chunking here.

  4. ✓ Emit loop, loss-timer, and ACK benchmarks, over one whole-connection set-up.
    • QuicConnection::emit draining a 1 MiB response's worth of queued 1-RTT frames into sealed packets, the per-packet emit loop, on a connection in the established state with application keys.
    • QuicConnection::setLossDetectionTimer against a sent_packets set of the size a /big response leaves in flight, to measure the walk the note says could be cached.
    • QuicConnection::processAck over a representative ACK frame, the inbound cost that dominates the datagram path for /big.

    Done: tests/h3bench/bench_emit.php builds one established connection the http3-test way — ReflectionClass without the constructor, public fields set directly, protected processAck reached by reflection with no setAccessible (that call is a no-op since PHP 8.1 and deprecated in 8.5) — and times all three. emit gets a whole response queued via flushStreams, snapshotted, and restored before each timed call; processAck likewise restores its in-flight set each call, since both mutate the state they run on. Measured on this box: emit ~19.6 ms for the 1 MiB response, ~20.6 us/packet across 954 packets; setLossDetectionTimer ~27 us over 950 in-flight; processAck ~216 us retiring 950. The finding: emit's ~20.6 us/packet is far above the ~6 us encodeShort costs on its own (item 2), and the gap is emit calling setLossDetectionTimer once per packet over an in-flight set that grows as the response ships — an O(n^2) walk over the send. setLossDetectionTimer measured alone at ~27 us for 950 in-flight is the same walk; caching its earliest deadline (the note's suggestion) would cut both it and the bulk of emit's per-packet overhead, and these baselines are what a cache change would be measured against. To keep the heavy emit call from paying the full warm-up, measure.php gained an optional warm-up count. Run with php tests/h3bench/bench_emit.php.

    Arc complete: every hotpath method the note names now has a benchmark under tests/h3bench/, runnable alone or together through bench_all.php, out of the pass/fail run_all.php. Two of the note's proposed wins turned out already done or not worth doing here — the send-buffer O(n^2) drain is fixed (item 3), and skipping the frame-encode switch saves nothing measurable (item 2); the one clear remaining win the numbers point at is caching the loss-detection timer's earliest deadline, which shows up in both setLossDetectionTimer and emit (item 4).

Decisions, as settled through the arc

  • Where the benchmarks live and how they run. They are timed, not asserted, and slow by design, so they must stay out of tests/run_all.php, which globs test_*.php and flags anything over 0.01 s as unintended I/O. The plan puts them in tests/h3bench/ as bench_*.php (the bench_ prefix keeps the runner from picking them up), with a bench_all.php that runs them and prints a table; each is also runnable on its own. If you would rather they sat outside tests/ entirely (a top-level bench/), say so. Resolved: kept in tests/h3bench/ as bench_*.php with bench_all.php; run_all stays 16 files green and never picks them up.
  • How the connection-stateful methods are set up. The last two items need a QuicConnection, which the constructor builds from a certificate and key and which then needs a response queued and some packets recorded as in flight. The plan stands up one throwaway self-signed cert at benchmark start and a shared helper that builds a representative connection (a 1 MiB stream queued, keys installed), reaching internal fields through reflection only where the class does not expose them. The open question is how faithfully to reproduce the real /big state versus keeping the set-up small; the plan leans toward the smallest set-up that exercises the same code, and will say in each benchmark what state it stood up. Resolved: no certificate — the connection is built with ReflectionClass without the constructor and only the read fields set, the way the http3 tests build one; each benchmark states the state it stands up.
  • Protected methods. encodeStream and processAck are protected. The plan reaches them with reflection to time the one method in isolation, and, where it is cheap, also times the public caller (encode, the datagram path) so the dispatch overhead the note mentions is visible too. Say if you would rather only ever go through the public entry points. Resolved: reflection reaches them (getClosure for encodeStream, invokeArgs for processAck), with no setAccessible, which PHP 8.5 deprecates; encode's public switch path is timed alongside encodeStream so the dispatch cost is visible.
  • Baselines now, tweaks after. This arc's deliverable is the benchmarks and the baseline numbers, not the optimizations. Once the baselines are in, the note's candidate wins (batching header-protection masks, caching the loss-timer earliest deadline, pooling per-packet allocations) can each be tried and measured against its baseline, as a follow-on. If you would rather fold the tweak-and-measure for a given method into the same item that benchmarks it, say which methods. Resolved: baselines only this arc; the numbers point at caching the loss-timer earliest deadline as the one clear remaining win, left as a follow-on.

Follow-on: caching the loss-detection timer

The one clear win the baselines pointed at, now implemented. setLossDetectionTimer used to walk every in-flight packet across all packet-number spaces to find the earliest ack-eliciting send time, and emit called it once per packet as a response shipped — an O(n^2) rescan over a 1 MiB send, item 4's ~20.6 us/packet. That walk is now cached.

  1. ✓ A cached earliest ack-eliciting send time on the connection, kept current incrementally.
    • Recording a sent packet folds it into the cache in O(1): a new packet is never older than the current earliest, so it only fills an empty cache (noteSentForLossTimer, called at the three points emit records a packet). This is what keeps the emit loop's per-packet timer updates from rescanning.
    • Removing a packet (an ack in processAck, a loss in detectAndRemoveLostPackets) cannot know in O(1) whether it took the earliest, so it marks the cache stale. setLossDetectionTimer recomputes the earliest — the old walk, now computeEarliestElicitingSend — only when the cache is stale, and reads it directly otherwise. The stale flag defaults on, so a freshly built or directly-populated connection recomputes on first use rather than trusting an unbuilt cache.

    Correctness gate: the whole test set stays 16 files green, including the rtt/pto, loss-recovery, and ack-processing tests that drive the timer directly. Results on this box, against item 4's baselines: setLossDetectionTimer ~22 us on the cold walk of 950 in-flight versus ~0.08 us reading the warm cache; and emit from ~20.6 to ~5.2 us/packet — ~19.6 ms down to ~5.0 ms for the 1 MiB response — since its ~950 per-packet timer calls are now cache reads rather than O(n) walks. emit's per-packet cost now sits near the ~6 us encodeShort seal that is its real floor. bench_emit.php gained a cold-walk row and a cached row so both are visible; run php tests/h3bench/bench_emit.php.

  2. ✓ A cache-versus-walk check of the processAck path, to see whether it hides a rescan like the timer did.

    It does not, mostly. processAck's cost is the per-acked-packet removal loop, and each packet's work (removeFromBytesInFlight, onPacketAckedForCwnd, the unset) is O(1), so retiring 950 packets at ~216 us is inherent, not a redundant walk to cache. The setLossDetectionTimer call it ends with is short-circuited by loss_time (a three-entry check) in the common case where packets remain below largest_acked, so it does not re-walk the in-flight set either.

    The one thing the check did turn up is in detectAndRemoveLostPackets, which processAck calls: it walked the whole in-flight set, skipping past every packet newer than largest_acked with a continue. Since sent_packets is in packet-number order, once the scan reaches one newer than largest_acked every later entry is newer too, so that is now a break — the recent in-flight tail above largest_acked is no longer scanned. All 16 test files stay green, including the loss-recovery tests that drive this method. Measured with a full 950-packet in-flight set and a low largest_acked, the scan runs in ~0.5 us versus ~26 us to walk the whole set; in steady state the saving is proportional to how many in-flight packets sit above largest_acked (the recently sent, not-yet-acked ones). It is a small, safe win rather than the timer's large one, which is the honest result of the check.

  3. ✓ A plain-English pass over the docblocks and comments of src/H3Listener.php, so the file the benchmarks exercise reads on its own terms.

    One documentation pass across all thirteen classes and the two stream-state enums, changing no behavior. Removed the development-phase references ("Phase 4", "Phase 11b", and the like) that meant nothing to a later reader, keeping the real "key-phase" protocol term. Gave each RFC a one-line gloss of what it covers on first mention (9000 transport, 9001 packet protection, 9002 loss and congestion, 8446 TLS 1.3, 5869 HKDF, 8017 RSA signatures, 8439 ChaCha20/Poly1305, 9114 HTTP/3, 9204 QPACK, 7541 HPACK) and expanded acronyms on first use. Corrected the many @param and @return lines that disagreed with the real signatures — returns that claimed void, int, or QuicPacket where the method returned a bool, a string, or an array, plus every @param mixed placeholder left by an earlier stub pass. Converted the last // comments to block form and wrapped every pre-existing over-80-column line. All 16 test files stay green, and the file now has zero phase references, zero @param mixed, zero // comments, and zero lines over 80 columns.

  4. ✓ Tie the per-method timings to real request shapes: for each case of the example-21 HTTP benchmark, count how many times each benchmarked method runs while the pure-PHP listener serves it, and multiply by that method's single-call cost.

    Two files under tests/h3bench/. The counting is done without adding anything to the shipping src/H3Listener.php: counting.php holds reusable subclasses of the QUIC connection, stream, and packet-key classes that override the timed methods to tally a call and then run the real one through parent. Because the real code reaches those methods through the object it holds ($this->emit(), $keys->seal(), $stream->takeForFrame()), a driver built from these subclasses counts every call, including the ones the code makes to itself. It is kept as a shared helper so a later test or benchmark can reuse it. The methods the code calls statically (encodeShort, decodeShort, QuicFrame::encode, encodeStream, decodeAll) cannot be overridden, so their counts are derived one-to-one from the counted ones and marked as derived.

    bench_workload.php builds each case's real HTTP/3 response bytes, drives them out through the paced flush-and-emit loop, and drives the request and the client's acknowledgements back in as decoded short packets, then reads the single-call microsecond figures from the other bench_ scripts' output and multiplies. It reports, per case: the response and inbound packet counts; the per-primitive call counts; the estimated microseconds (single-call cost times count) for the primitive methods; and, separately, the call counts of the whole-operation methods (emit, flushStreams, drain, processAck, the loss timer, the lost-packet scan), which are not multiplied because their timed cost is a whole-1-MiB or 950-acked figure that already sums the primitives. The estimate rows are each one method's own cost and are not added together, since encodeShort contains seal and header-protection, decodeShort contains open, and QuicFrame::encode contains encodeStream. All 16 test files stay green; the new scripts live under the bench_ name the test runner skips.

    Follow-up: the counting keys were copied with ReflectionProperty::setAccessible, which PHP 8.5 deprecates because it has had no effect since 8.1; the reflection get and set work without it, so the two calls were dropped and the run is quiet again on 8.5.

  5. ✓ Speed up the per-packet nonce build, which runs once per seal and once per open. The old build exclusive-or'd the 64-bit packet number into the initialization vector one byte at a time in a twelve-step loop; the new one leaves the top four bytes alone and exclusive-ors the low eight in a single bulk string operation, which PHP runs in C.

    Measured with a new bench_nonce.php under tests/h3bench/, which also keeps the byte-loop form so the comparison stays reproducible after the source moved on, and checks every candidate against the shipping method across two thousand random vector-and-number pairs before timing. On this machine the byte loop ran near 1.1 us per build and the new bulk exclusive-or near 0.25 us through the real method call, about four and a half times quicker. A further variant that caches the vector's tail as an integer measured near 0.13 us, but it needs two stored fields and, across a whole 1 MiB response's worth of nonce builds, saves only about a tenth of a millisecond more than the bulk exclusive-or, so it was left as a documented alternative in the benchmark rather than shipped. All 16 test files, including the ones that drive real seal and open, stay green.

  6. ✓ Cut the send path's openssl_encrypt calls by batching header protection. Each outgoing 1-RTT packet made two: an aes-128-gcm seal, which has its own nonce, header, and tag and so cannot be shared, and an aes-128-ecb header-protection mask over a 16-byte sample. AES-ECB encrypts each 16-byte block on its own, so a whole emit()'s worth of samples concatenated and encrypted in one call give the masks back in order — one call in place of one per packet. Header protection is a length-preserving exclusive-or applied after sealing, so every length-based send gate (anti-amplification, congestion window) and the sent-packet bookkeeping run on the sealed bytes unchanged, and only the final wire output needs the mask; that is what lets the mask step move to the end of emit().

    encodeShort was split into encodeShortSealed, which seals and builds the header and returns the sample, and a static applyShortHeaderProtection, which does the exclusive-or; encodeShort now chains the two and behaves exactly as before. QuicPacketKeys gained headerProtectionMasks, which for the AES suites does the single ECB call and for ChaCha20 loops the single-sample method, since that suite derives each mask from a per-packet keystream nonce that cannot be concatenated. emit() seals each 1-RTT packet, keeps the existing bookkeeping, records the sample, and after the send loop masks the whole batch in one call. Long-header handshake packets keep their inline per-packet masking, and the receive side keeps headerProtectionMask per packet because datagrams arrive one at a time. The batched masks were checked byte-for-byte against the per-packet masks, the batched wire byte-for-byte against one-shot encodeShort, and every batched packet was decoded back to its payload, for AES and for ChaCha20, in a new tests/http3/test_hp_batch.php. On this container a 30-packet batch's header protection dropped from about 24.3 us done per packet to about 0.85 us batched, so a 30-packet emit() goes from about 60 openssl_encrypt calls to about 31; the 30 seals are untouched. bench_emit.php gained a per-packet and a batched header-protection row at the flush-budget batch size, and bench_codec.php gained an encodeShortSealed row. bench_workload.php now models the batched send path: the batched call is counted once per emit (33 for the 1 MiB case) and the per-packet headerProtectionMask row is the receive-side count, with encodeShortSealed replacing encodeShort as the per-packet send primitive. All 17 test files stay green.

  7. ✓ Implemented the QUIC ACK-Frequency extension (draft-ietf-quic-ack-frequency-14), which lets a data sender ask its peer to acknowledge less often. The reverse-path acknowledgment stream is what the benchmark exposed as the inbound decodeShort cost, but the reason to carry it is real browser traffic: on asymmetric and mobile links the acknowledgment volume constrains forward throughput, and on any high-rate connection it burns processing on both ends. None of that shows on a near-zero-RTT loopback run, so this is measured against real clients over a delayed path, not the localhost numbers.

    The listener advertises the min_ack_delay transport parameter (0xff04de1b, 1000 microseconds) and an explicit max_ack_delay (25 ms) so the draft's min-not-greater- than-max constraint holds. It parses both from the peer; a peer min_ack_delay above its own max_ack_delay declines the extension for that connection rather than closing it, which keeps a sound connection up. The ACK_FREQUENCY frame (type 0xaf, a two-byte variable-length integer on the wire) and IMMEDIATE_ACK frame (type 0x1f) are added to the frame codec; both are already ack-eliciting under the existing rule. On the honor side, a received ACK_FREQUENCY updates the threshold, requested max ack delay, and reordering threshold, ignoring any frame whose sequence number is not newer; a received IMMEDIATE_ACK forces the next acknowledgment. The acknowledgment decision moved into shouldSendAppAck, which keeps the previous behavior of acknowledging whenever one is pending until an ACK_FREQUENCY frame is received, so existing client interoperability is unchanged, and afterward acknowledges once more than the requested threshold of ack-eliciting packets have arrived, once the requested delay has passed, or on an IMMEDIATE_ACK. On the send side, once established and only if the peer advertised the extension, the listener sends one ACK_FREQUENCY frame asking for an Ack-Eliciting Threshold of 10, a Requested Max Ack Delay taken from the smoothed round-trip time bounded by 25 ms above and the peer's min_ack_delay below, and a Reordering Threshold of 1 so loss detection stays prompt. A requested delay below this endpoint's own min_ack_delay is raised to that floor rather than treated as an error. A new tests/http3/test_ack_frequency.php covers the transport-parameter advertisement and parse, the min-above-max decline, the frame codec round-trip and the two-byte wire type, the newer-sequence and floor handling in the honor path, the shouldSendAppAck decision across default, threshold, delay, and immediate cases, and the send gate staying silent without peer support and sending exactly one frame with support. All 18 test files stay green.

  8. ✓ Made the ack-frequency negotiation observable, since its value can only be confirmed against real clients over a delayed path, not the loopback numbers. stats() gained an ack_frequency block reporting whether the peer advertised the extension and its min_ack_delay, whether this endpoint sent an ACK_FREQUENCY and with what threshold and requested delay, and a count of received packets that carried an acknowledgment. Divided by packets_sent that count is the peer's acknowledgment cadence: pointing a browser at a big response over an added-delay path and reading this ratio shows directly whether the browser honored the request. A new stats-block check and a recorded-delay check were added to test_ack_frequency.php. All 18 test files stay green.
