atto — H3 Listener Hotpath Benchmarks

Started 07-Jul-2026. The Post v2 note h3-status-and-bottlenecks.html lists, per method, where the pure-PHP HTTP/3 listener spends its time on a 1 MiB response, but it reported those numbers without a test anyone can run to reproduce or inspect them. This arc builds that missing measurement: for each hotpath method the note names, a benchmark that calls that one method in isolation over many iterations and reports its per-call cost. The point is twofold: reproduce the note's per-method figures so they can be inspected directly, and give each method a stable baseline so a later code tweak can be measured against it — run the benchmark, change the method, run it again, and read the difference. These are benchmarks, not the pass/fail tests: they stand up no H3 website and open no socket, they exercise a single method with representative inputs, and they are timed rather than asserted.

Two things already noticed while reading the code, worth carrying into the work rather than trusting the note blindly. The substr-chunking fix the note lists as a future win is already in place: QuicStream advances a send_buf_off offset instead of mutating the buffer, so takeForFrame's benchmark will measure the current offset-based cost, not the old O(n^2) shift. And the QuicConnection constructor takes a certificate and key, so the connection-stateful methods need a throwaway cert plus a representative response queued before they can be called; that set-up is the bulk of the work for the last two items.

  1. → The benchmark runner and the AEAD benchmarks — the largest single cost (~36 ms, about a third of a /big response) and the cleanest to isolate.
    • A small bench helper that warms a method up, times a fixed number of calls with hrtime, takes the median over a few repetitions to smooth noise, and reports nanoseconds per call and calls per second. A bench_all.php that runs the benchmarks and prints one table.
    • Benchmarks for seal, open, and headerProtectionMask on a real key set built from QuicPacketKeys::fromInitialDcid (AES-128-GCM, the cipher the note measured), with a ~1100-byte payload and a 16-byte header-protection sample, the sizes the note used.
    • Confirm the per-call figures land near the note's ~4.2, ~8, and ~2.6 us; where they differ, say so plainly, since the box and the openssl build both move these numbers.
  2. Frame and packet encode/decode benchmarks.
    • QuicFrame::encode on a STREAM frame (its hot case, which dispatches into encodeStream), and decodeAll over a buffer of the frames a response and its ACKs carry.
    • QuicPacket::encodeShort and decodeShort over a short-header packet, using the key set from item 1 so the AEAD and header-protection steps they call are exercised as they are in the real path.
  3. Stream-chunking and emit-loop benchmarks, the first that need object state stood up.
    • QuicStream::takeForFrame over a 1 MiB send_buf, drained in the frame-sized chunks a /big response uses, to measure the current offset-based cost.
    • QuicConnection::emit and flushStreams over a connection built with a throwaway self-signed cert and a 1 MiB response queued on a stream, so the per-packet bookkeeping the note flags is measured on a realistic queue.
  4. Loss-timer and ACK benchmarks.
    • QuicConnection::setLossDetectionTimer against a sent_packets set of the size a /big response leaves in flight, to measure the walk the note says could be cached.
    • QuicConnection::processAck over a representative ACK frame, the inbound cost that dominates the datagram path for /big.

Open decisions to settle before coding

  • Where the benchmarks live and how they run. They are timed, not asserted, and slow by design, so they must stay out of tests/run_all.php, which globs test_*.php and flags anything over 0.01 s as unintended I/O. The plan puts them in tests/h3bench/ as bench_*.php (the bench_ prefix keeps the runner from picking them up), with a bench_all.php that runs them and prints a table; each is also runnable on its own. If you would rather they sat outside tests/ entirely (a top-level bench/), say so.
  • How the connection-stateful methods are set up. The last two items need a QuicConnection, which the constructor builds from a certificate and key and which then needs a response queued and some packets recorded as in flight. The plan stands up one throwaway self-signed cert at benchmark start and a shared helper that builds a representative connection (a 1 MiB stream queued, keys installed), reaching internal fields through reflection only where the class does not expose them. The open question is how faithfully to reproduce the real /big state versus keeping the set-up small; the plan leans toward the smallest set-up that exercises the same code, and will say in each benchmark what state it stood up.
  • Protected methods. encodeStream and processAck are protected. The plan reaches them with reflection to time the one method in isolation, and, where it is cheap, also times the public caller (encode, the datagram path) so the dispatch overhead the note mentions is visible too. Say if you would rather only ever go through the public entry points.
  • Baselines now, tweaks after. This arc's deliverable is the benchmarks and the baseline numbers, not the optimizations. Once the baselines are in, the note's candidate wins (batching header-protection masks, caching the loss-timer earliest deadline, pooling per-packet allocations) can each be tried and measured against its baseline, as a follow-on. If you would rather fold the tweak-and-measure for a given method into the same item that benchmarks it, say which methods.
