atto — Development process for LLM co-implementers

How Chris Pollett likes development to run, written for an LLM working with him on atto. The conventions here are shared with his other projects; what is specific to atto is the way changes get in and out of the repository, called out in the first section. When something below says "always" or "never," treat it as a hard rule, not a preference.

Round counter. Keep a count of rounds since you last re-read this file. A round is one exchange: Chris says something, you respond. Echo the current count in every response so it is visible. When the count reaches 10, re-read this whole file before writing your next response, then reset the count to 0 and tell Chris you re-read it. This is a memory aid: it keeps these conventions fresh over a long session instead of letting them drift. The same counter runs in Yioop's copy of this file.

How changes move (atto-specific)

atto lives on GitHub at github.com/cpollett/atto. There is no local gold mirror or gold/work dance. The loop is simply:

  1. The LLM prepares a change and delivers it as a patch.
  2. Chris reviews it and, if good, pushes it to GitHub.
  3. The LLM re-clones the repository to resync before the next change.

So nothing is "in" until Chris has pushed it and the LLM has re-cloned. Never assume a delivered patch is committed; work from a fresh clone each round. Cut patches with git diff (or git add -A; git diff --cached when new files are involved so they appear), and before delivering, dry-run the patch on a throwaway clone with git apply --check so it is known to apply cleanly. Name a patch for what it does in plain terms — cooperative-fibers-and-dav-auth.patch, not a sequence number or a plan-item number.

Coding conventions (shared across projects)

  • 80 columns. No line over 80 characters. When a patch touches a file, fix every pre-existing long line in it too, not only the new ones (autogenerated data files are the one exception).
  • Braces always. Every if, else, and loop body is braced, even one-liners.
  • Block comments only. Use /* ... */; never //.
  • No dead code. If a change makes nearby code a no-op or pointless, delete it and say so, rather than leaving it.
  • No magic numbers. Every literal — a size, a limit, a timeout, a conversion factor — gets a named constant with a docblock. Grep for an existing constant to reuse before adding one.
  • No single-letter variables except loop indices i, j, k, in both PHP and any JavaScript.
  • No type or unit suffixes on names. Not _arr, _str, _int, _obj, _bool, _list, and not unit suffixes like _secs, _ms, _bytes. Put the unit in the docblock instead: HEARTBEAT_POLL, not HEARTBEAT_POLL_SECS. (_info, _max, _name read as words and are fine.)
  • Docblocks everywhere. Every class header and every method has a docblock whose first sentence a layperson can read: what it does and why. Technical specifics — an RFC number, an encoding — come after, in plain terms. An RFC citation standing in for an explanation counts as undelivered code.

Keeping the server files self-contained

The server sources (WebSite.php, MailSite.php, and the others) are deliberately decoupled: they carry no framework or config references and read no outside constants, so atto stays a set of single files a reader can follow end to end. Keep any conversion constants local to the file, named in an X_PER_Y shape with the units spelled out in the docblock. Know decimal versus binary and do not conflate them: PHP's memory_limit suffix M is 1024×1024, not a decimal million, so a decimal metric helper must not be used to read it.

Tests

atto's tests are standalone scripts under tests/<subsystem>/test_*.php, run by php tests/run_all.php (optionally filtered, e.g. php tests/run_all.php http3). Each script asserts with a small helper, prints a Tests run: / Tests passed: summary, and exit()s non-zero on any failure — that exit status is the pass/fail signal the runner reads. Follow that same contract for a new test. Pull pure logic out where you can so it can be checked on its own without a live server. Flag any test that takes more than about 0.01 seconds — it usually means unintended I/O. Call the data a test stands up its "test set-up" or "test data," never a "fixture."

The audit before every patch

Run these as separate checks and report PASS or FAIL for each; a patch is not ready until all pass:

  1. php -l on every touched file.
  2. Every class and every method it adds has a docblock.
  3. No new single-letter PHP variables except i, j, k.
  4. No banned identifier suffixes (the type/unit ones above).
  5. No // comments.
  6. No new single-letter JavaScript variables except i, j, k.
  7. No line over 80 columns in a touched file (e.g. awk 'length > 80' over each).
  8. Every if body is braced.
  9. No banned words and no dead code.

Verify before you state

  • Read the actual code before asserting how it behaves. If it is not in view, say so — never present a guess as a mechanism.
  • Grep for a named helper before claiming it does not exist. Chris wrote the codebase; a function he refers to by name usually is there.
  • Run it. Start the server, exercise the routes, and confirm the behavior before presenting — do not describe output you have not seen. When a change might make nearby code slower or redundant, say so plainly rather than hiding it.

Asking questions

  • When you need Chris to decide something, ask in plain prose. Do not use option or button prompts, multiple-choice pickers, or any similar tap-to-answer control. Lay out the choices and their trade-offs in the text of your response and let him answer in his own words.

Delivering a patch

  • Prefer one larger, complete, testable patch over a run of tiny ones. A cohesive feature ships as a single patch; fold a small fix into the next larger one rather than cutting a patch for it alone.
  • Present the pass/fail audit, a plain-English sentence or two on what the patch does, and exactly what to run and look for to check it.
  • Do not check off a devlog item until Chris says it is done. Marking work complete before he confirms it is not welcome.

Devlog

Work is logged under devlog/ in dated arc folders named YYYY-MM-DD Arc Name/, each with an HTML plan document (the other folders beside this file are the pattern to copy). The devlog is append-only: a finished item keeps its checkmark and its one- or two-line note as a permanent record, and a later hotfix goes under the current arc's plan rather than starting a new dated folder. Every code patch also carries the matching plan update — the next item checked off, with a short note on how it was done — committed inside the same patch. Before presenting, check that the plan's numbering is internally consistent and that what is marked done lines up with what is next. Try to follow the rules for traditional outlining call for outlines to begin with Roman numerals (I, II, III) and to proceed through letters (A, B, C, and a, b, c) and Arabic numerals (1, 2, 3) Plan upper roman items should be always titled with an action that can be carried. within. Internal ordered lists can be steps to carry that out. Plan items shoudl avoid general discussion. They are always targeted to concrete actions and should be as succinct as possible.

Words to avoid

These read as offensive or as hedging to Chris; do not use them, in code or in conversation:

  • Never: harness; ctor (write "constructor"); rails; sidecar (say ".flags/.date companion files"); fixture (say "test set-up"); nit; and proof/proven/"I proved" for empirical results (say "evidence").
  • Never: to be honest, honestly, genuinely, actually, "the honest finding," "straight with you." They imply everything else was not, which is not the tone to take.
  • Never open with a throat-clearing announcement that you are about to say something plainly, honestly, or directly, and then say it. Banned lead-ins include "what I want to be plain about is," "the one thing I want to be plain about," "I want to be plain," "let me be plain," "to be plain," "to be clear," "I'll be direct," "let me be direct," "let me be straight," and every close variant. This phrasing is distressing and adds nothing: just say the thing, with no lead-in.

Running things in this environment

  • Chris's shell is tcsh; write commands and scripts in tcsh syntax (no bash-style loops; single-quote any string containing !).
  • Start an atto server so it cannot hang the session: give it a self-terminating timeout that cannot be trapped, e.g. timeout -s KILL 12 php index.php 14 </dev/null >/tmp/log 2>&1 &, and give curl a --max-time so it cannot wait forever.
  • Free a busy port before a fresh run with fuser -k 8080/tcp; a stale server left on the port will answer with the old code and hide the change under test.
